# renderer

The native renderer process for the Arc framework.

Arc spawns this binary as a subprocess to display application UI using the
platform's native webview. All window management, navigation, and IPC is
driven entirely by the parent Go process over a persistent IPC connection.
The renderer has no application logic of its own — it is a display process,
nothing more.

---

## Overview

```text
Arc (Go)  ──── IPC ────  renderer  ──── native webview ──── JS
```

On startup the renderer connects to an IPC channel opened by the Arc host
process and waits for commands. The host drives everything — creating the
window, loading content, and exchanging IPC messages with JavaScript. The
renderer forwards all inbound JS messages back to the host over the same
channel.

```text
Arc (Go)                        renderer                      JS
   │                               │                           │
   ├── WindowCreate ──────────────>│                           │
   │<──────────────────── Ready ───┤                           │
   │                               │                           │
   ├── LoadFile ──────────────────>│                           │
   │                               ├── navigate ─────────────>│
   │                               │                           │
   ├── PostText(channel, payload) >│                           │
   │                               ├── ipc.on(channel) ──────>│
   │                               │                           │
   │                               │<── ipc.post(channel) ────┤
   │<── IpcText(channel, payload) ─┤                           │
   │                               │                           │
   ├── Quit ──────────────────────>│                           │
   │<──────────────────── Closed ──┤                           │
```

## IPC Transport

The renderer communicates with the Arc host process over a persistent
full-duplex channel using a length-prefixed binary framing protocol. All
commands and events flow over this channel for the lifetime of the process.

| Platform | Transport   | Path format               |
|----------|-------------|---------------------------|
| Linux    | Unix socket | `/tmp/arc-<id>.sock`      |
| macOS    | Unix socket | `/tmp/arc-<id>.sock`      |
| Windows  | Named pipe  | `\\.\pipe\arc-<id>`       |

The channel ID is generated by the Arc host process at spawn time and
passed to the renderer via the `--channel` flag. The renderer constructs
the platform-specific path from the ID internally.

```bash
renderer --channel 1234        # resolves to \\.\pipe\arc-1234   (Windows)
                               # resolves to /tmp/arc-1234.sock  (Linux / macOS)
```

Logging is disabled by default and can be enabled with `--logging`:

```bash
renderer --channel 1234 --logging
```

All log output goes to stderr.

### Commands (Host → Renderer)

| Command        | Description                                  |
|----------------|----------------------------------------------|
| `WindowCreate` | Create and initialise the native window      |
| `LoadFile`     | Navigate to a local file via `ui-ipc://app/` |
| `LoadHTML`     | Load inline HTML via `ui-ipc://app/`         |
| `LoadURL`      | Navigate to an external URL                  |
| `Eval`         | Execute JavaScript in the current page       |
| `SetTitle`     | Update the window title                      |
| `SetSize`      | Resize the window                            |
| `PostText`     | Send a text message to a JS IPC channel      |
| `PostBinary`   | Send a binary message to a JS IPC channel    |
| `Quit`         | Terminate the renderer                       |

### Events (Renderer → Host)

| Event       | Description                                   |
|-------------|-----------------------------------------------|
| `Ready`     | Window and webview are initialised and ready  |
| `Closed`    | Window was closed by the user                 |
| `IpcText`   | JS posted a text message on a channel         |
| `IpcBinary` | JS posted a binary message on a channel       |

## Custom Scheme (`ui-ipc://`)

Local content is never loaded from `file://`. Instead, the renderer
registers a custom `ui-ipc://` scheme with the webview and intercepts
all requests internally. This keeps content sandboxed and gives the
renderer full control over what is served.

Two authorities are in use:

### `ui-ipc://app/` — content serving

Serves the application's HTML, CSS, JavaScript, and assets.

- **`LoadHTML`** — any request to `ui-ipc://app/` returns the inline HTML
  string held in memory.
- **`LoadFile`** — requests for `ui-ipc://app/<path>` are resolved against
  the directory of the entry file on disk. Path traversal outside that
  root is blocked. MIME types are inferred from file extensions.

### `ui-ipc://host/message/<channel>/<token>` — binary slot retrieval

When the host sends a binary message to JavaScript, the renderer stores
the payload in an in-memory slot keyed by `channel:token` and posts a
lightweight JSON notification to the webview via `postMessage`. The JS
shim then fetches the binary payload from this URL, clearing the slot once
consumed.

## JavaScript IPC

The renderer injects a lightweight IPC shim into every page before any
script runs. This gives JavaScript a consistent interface for communicating
with the Arc host process regardless of platform.

```js
// Send a text message to the host
ipc.post("channel", "hello")

// Send binary data to the host
ipc.post("channel", new Uint8Array([0x01, 0x02, 0x03]).buffer)

// Receive a message from the host (text or binary)
ipc.on("channel", (payload) => {
    if (payload instanceof ArrayBuffer) {
        // binary
    } else {
        // text
    }
})

// Unregister
ipc.off("channel")
```

All messages are routed transparently through the renderer to the Arc host
process. The renderer does not handle or inspect message content.

## Building

Requires CMake 3.22+ and a C++20 compiler.

### Windows

vcpkg is used to pull in the WebView2 SDK. Bootstrap it directly into the
renderer folder, then invoke CMake with the vcpkg toolchain file.

```bat
git clone https://github.com/microsoft/vcpkg
.\vcpkg\bootstrap-vcpkg.bat
.\vcpkg\vcpkg.exe install
```

Then configure and build:

```bat
cmake -B build -DCMAKE_TOOLCHAIN_FILE="vcpkg/scripts/buildsystems/vcpkg.cmake"
cmake --build build
```

### macOS

No additional dependencies. WKWebView is provided by the system.

```bash
cmake -B build
cmake --build build
```

### Linux

Install the WebKit2GTK and GTK development packages, then build.

```bash
sudo apt install build-essential libwebkit2gtk-4.1-dev libgtk-3-dev cmake
cmake -B build
cmake --build build
```

## Structure

```text
renderer/
├── include/
│   ├── host_channel.h        # host channel, command, and event types
│   └── logger.h              # cross-platform logging utility
├── src/
│   ├── host_channel_unix.cpp # Unix socket IPC transport (macOS/Linux)
│   ├── host_channel_win.cpp  # Named pipe IPC transport (Windows)
│   ├── logger.cpp            # stderr logger, enabled via --logging
│   ├── main.cpp              # entry point — parses flags, runs command loop
│   └── browser/
│       ├── shared/           # Cross-platform browser interfaces & helpers
│       │   ├── mime.h        # MIME type resolution
│       │   ├── str_escape.h  # JS string escaping
│       │   ├── types.h       # Common browser types
│       │   └── webview.h     # Platform-agnostic WebView interface
│       ├── win/              # Windows — WebView2 implementation
│       ├── mac/              # macOS — WKWebView implementation
│       └── linux/            # Linux — WebKit2GTK implementation
├── vcpkg/                    # vcpkg clone (Windows only, not committed)
├── vcpkg.json                # vcpkg manifest — declares WebView2 dependency
└── CMakeLists.txt
```

The IPC layer is split into two distinct responsibilities:

- `host_channel_*.cpp` — the channel between the renderer and the Arc host process (named pipe / unix socket)
- `browser/*/ipc.*` — delivery of outbound messages from the renderer into the webview's JavaScript context
